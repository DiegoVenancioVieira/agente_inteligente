API de Intenção

Recebe o texto livre do cidadão e devolve o assunto do 1doc correspondente, para o sistema chamador localizar o formulário e montar o card. Substitui o Dialogflow.

📊 Relatório desta API 🏛 Órgãos (JSON) ⚙️ OpenAPI

Como funciona

Não usa LLM. É embedding (bge-m3) mais similaridade de cosseno contra os 1.888 sinônimos cadastrados. Isso não é economia — é o desenho correto para a tarefa:

  • Não inventa assunto. A resposta sai sempre da lista do 1doc. Uma LLM generativa poderia devolver um assunto que não existe, e o POST seguinte quebraria.
  • ~300 ms, sem entrar na fila do modelo de linguagem (que processa uma resposta por vez na CPU).
  • Entende paráfrase. O Dialogflow casava o texto contra o sinônimo literal. Aqui, “preciso dividir o IPTU em várias vezes” encontra parcelamento de iptu sem que ninguém tenha cadastrado essa frase.
  • Assume a ambiguidade em vez de escolher no chute.

Experimente

POST /intent público

{
  "text":  "quero parcelar meu iptu",   // obrigatório, 1..500 caracteres
  "orgao": "semfaz",                    // opcional — ver Ambiguidade
  "top_n": 3                            // opcional, 1..10 (padrão 3)
}

Em ambiguo e nao_identificado, intent vem null de propósito. Nunca use o primeiro candidato nesses casos — é exatamente a situação em que ele não é confiável.

Ambiguidade — leia antes de integrar

88 sinônimos do cadastro apontam para mais de um assunto. Isso é um teto: nenhum modelo resolve, porque a informação não está no texto. O Dialogflow também não resolvia — apenas escolhia um em silêncio.

sinônimoaponta pararesolve com orgao?
nota fiscal23 assuntos (um por secretaria)✅ sim
mandado judicial20 assuntos✅ sim
parcelar meu iptuparcelamento × reparcelamento❌ não — mesmo órgão

Mande sempre o orgao se o sistema já souber a secretaria do atendimento. Ele restringe a busca e elimina os dois maiores casos. Siglas em GET /intent/orgaos.

As 8 colisões que o orgao não resolve

São distinções de negócio reais, dentro da mesma secretaria:

  • parcelamento × reparcelamento — de IPTU, de ISS e de TLF (SEMFAZ)
  • isencao de iptu × isencao de iptu servidor (SEMFAZ)
  • isencao de itbi × deferimento parcial da isencao do itbi (SEMFAZ)
  • autorizacao de demolicao × certidao de demolicao (EMURB)
  • autorizacao para obras de infraestrutura — ligações de água/esgoto × obras em geral (EMURB)
  • nota fiscal sms × nota fiscal sms saude — duplicata de cadastro, não distinção de negócio: vale corrigir no 1doc

“Quero parcelar meu IPTU” é genuinamente indistinguível: depende de o cidadão já ter um acordo ativo. Há duas saídas — card de desambiguação, ou o chamador decide por regra de negócio (consultando se existe parcelamento em vigor).

“Auxílio funeral” merece atenção. Casa com auxilio funeral (SEPLOG, servidor municipal) e auxilio funerario (SEMFAS, assistência social) — públicos diferentes. Chutar aqui abriria o formulário errado para uma família enlutada. Por isso a API devolve ambiguo em vez de escolher.

Operação

Quando a lista de assuntos do 1doc mudar

Atualize sources/intents-1doc.json e rode o reindex. Leva ~2 minutos (um embedding por sinônimo) — é o único passo lento, e não deve rodar a cada requisição.

curl -X POST http://192.168.0.118:8100/intent/reindex \
     -H "Authorization: Bearer $ADMIN_TOKEN"

# -> {"status":"ok","sinonimos":1888,"assuntos":756,"segundos":122.4}

Depois de um redeploy

Confira GET /health. Se intent.sinonimos vier 0, o índice não foi carregado e o reindex precisa rodar — enquanto isso o /intent responde 503.

Configuração

INTENT_THRESHOLD0.70abaixo disso → nao_identificado
INTENT_MARGIN0.04diferença 1º–2º abaixo disso → ambiguo
INTENT_RATE_LIMIT_PER_MIN600limite do /intent

Os dois primeiros foram medidos, não chutados — a metodologia está no relatório.

Limite de requisições. O /ask (chat do cidadão) limita 20/min por IP. Numa integração servidor-a-servidor todas as chamadas chegam do mesmo IP, e esse limite derrubaria a API na 21ª pergunta do dia. Por isso o /intent tem limite próprio (600/min). Se a API for exposta fora da rede interna, troque o limite por IP por autenticação com chave por cliente.
Base de conhecimento: sources/intents-1doc.json — 758 assuntos, 1.888 sinônimos, herdados do json_normalizado.json que alimentava o Dialogflow. Dados medidos e limitações conhecidas no relatório desta API.
